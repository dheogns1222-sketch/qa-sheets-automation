# 수식 기반 → 코드 기반: 왜 옮겼나

원본 시스템은 Google Sheets 수식만으로 구현되어 4개 프로젝트에 실배포됐습니다. 실사용 규모는 단일 리더 문서 기준 IFERROR 8,000회 이상 — 수식 시스템으로서는 한계까지 간 구조였습니다. 코드 판으로 옮긴 이유는 세 가지입니다.

## 1. 유지보수성

- **수식**: `IMPORTRANGE + REGEXEXTRACT` 같은 중첩 수식은 작성자만 고칠 수 있습니다. 파견이 끝나면 시스템도 사실상 끝났습니다 (사용법 문서 4종으로 버텼지만, 구조 변경은 불가능).
- **코드**: 모듈 단위 책임 분리 + JSDoc — 다음 사람이 diff로 변경을 추적할 수 있습니다.

## 2. 테스트 가능성

- **수식**: "이 셀이 왜 0%지?"를 검증할 방법이 눈으로 보는 것뿐이었습니다.
- **코드**: 집계 로직을 `lib/core.js`(Sheets API 무의존)로 분리해 `node --test`로 22개 단위 테스트를 돌립니다. 0 나누기, 빈 시트, 잘못된 날짜 포맷 같은 엣지가 회귀 테스트로 고정됩니다. **QA가 만든 도구가 정작 테스트 불가능하다는 모순의 해소.**

## 3. 수식으로 불가능했던 것

| 요구 | 수식의 한계 | 코드 판 |
|---|---|---|
| 보고서 자동 "발송" | 불가능 (생성까지만) | 시간 트리거 + MailApp/웹훅 |
| 권한 오류의 명시적 처리 | IMPORTRANGE 권한 거부 시 #REF! 침묵 | try/catch 후 오류 리포트 시트 기록 |
| 타임존 고정 | TODAY()가 문서 설정에 암묵 의존 | 스크립트 속성으로 명시 |
| 시트 구조 변경 감지 | 열 삽입 한 번에 수식 전체 붕괴 | 헤더 이름 기반 열 탐색 |

## 옮기며 지킨 것

- **결과값·심각도 라벨 체계** (Pass/Fail/Block/N/A/No Run · Highest~Lowest): 실무 표준 그대로.
- **권한 3계층과 색상 규약**: 운영에서 검증된 규칙은 코드에서도 상수로 유지 (`StyleGuide.js`).
- **"0건이면 −(대시) 표기"** 같은 보고 문화의 디테일: 고객사가 읽는 방식은 도구가 바뀌어도 유지되어야 합니다.
